提交规范

开发过程

  • master分支始终保持可发布状态，即master分支里的代码都是完整的功能、bug修复提交，随时发出去都没问题。
  • 不是当前就要发布的代码 必须 切换到分支上开发，不然如果有bug修复需要紧急发布，就有可能把当前还不能发布的代码也发布出去。即除了 小 的bug修复，一般新功能和 大 的bug修复必须在分支上进行。
  • 开发完成后，经测试可发布后，再把修改合并回主干（master）分支。
  • 同一功能 不要 一个个文件分开提交，请一起提交。
  • 多个功能点或bug修复，除非是相关的，请分成多次提交。这样如果有个bug/功能点的代码有问题，只需要还原有问题的bug/功能点。

分支及提交信息命名规范

  • 提交信息必须以下列类型标识开头后面跟:（英文冒号+半角空格），例如：refactor: 代码按照PSR-2规范格式化。
  • 创建分支建议以下列类型标识开头后面跟/（斜杠），例如feat/weixin_login。
  • 提交信息不包括提交类型及空格必须 >= 6 个字符。

  • 如果一次提交中包含多个相关功能点或bug修复等，提交信息中只需要出现一次提交类型，可第一行概述下本次提交的内容，然后空一行，再分行列出详细的提交修改。多行信息提交示例：

  • 提交日志信息的第一行末尾不用跟句号。

提交类型

  • feat: 新功能(feature)
  • change: 功能变更，例如查询时间范围由1个月变成3个月
  • remove: 删除功能
  • fix: 修复缺陷
  • refactor: 重构，并未修复缺陷或添加功能

进入本机存放代码的目录，右键菜单选择 Git Commit，如下图

按照下图箭头所示的顺序操作

  • 勾选Amend Last Commit(修改上次提交)复选框

  • 在Message(消息输入框)中修改上次提交信息
  • 点击Commit(提交)按钮
